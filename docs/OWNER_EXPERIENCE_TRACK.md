# Owner Experience Track

Last updated: 2026-03-28 UTC

## Purpose

This document tracks the owner-facing value direction of the `eeepc` system.

It exists to make an important boundary explicit:

the role of maintainers is to build the base architecture, policies, and runtime
capabilities that enable owner-facing outputs.
The steady-state goal is for the running system itself to generate and improve
those outputs.

## Scope

Owner experience may include:

- practical workflow help,
- terminal-native interfaces,
- dashboards and status views,
- useful utilities,
- research summaries,
- visual or interactive artifacts,
- games, graphics, and demoscene-style experiments,
- other outputs the owner can judge, enjoy, and refine.

## Architecture Framing

This track is not primarily about maintainers manually building TUI apps or
creative software by hand.

This track is about making the base system able to:

- understand what the owner values,
- create bounded outputs for the owner,
- evaluate which outputs are useful,
- improve those outputs over time,
- stay within constrained-host limits while doing so.

## Owner Value Categories

### 1. Practical Utility

Examples:

- workflow helpers,
- host and project summaries,
- diagnostics made readable for the owner,
- small tools discovered through runtime research.

### 2. Interaction And Visibility

Examples:

- terminal dashboards,
- operator-facing status views,
- resource visualizations,
- better interaction layers generated by the runtime.

### 3. Creative And Evaluable Artifacts

Examples:

- generated graphics,
- small games,
- demoscene-style experiments,
- visual or audio outputs,
- playful or artistic prototypes that can be assessed and improved.

## Maintainer-Owned Work In This Track

Maintainers should focus on:

- runtime capability scaffolding,
- policy boundaries,
- evaluation hooks,
- artifact storage and retention rules,
- portability constraints,
- recovery and rollback for failed experiments,
- documentation of what kinds of outputs are allowed.

## Runtime-Owned Work In This Track

The running system should increasingly own:

- generating the outputs,
- revising them,
- reacting to feedback,
- deciding which directions are worth deeper iteration,
- learning owner preferences from evidence and use.

## Evaluation Signals

Owner-facing outputs should be evaluated by some combination of:

- direct owner feedback,
- repeated usage,
- measured utility,
- perceived clarity,
- novelty or delight,
- constrained-host affordability,
- maintainability of the generation path.

## First Architecture Tasks For This Track

Near-term architectural work should focus on:

- defining how owner feedback is captured,
- defining where generated artifacts live,
- defining evaluation metadata for owner-facing outputs,
- defining cost ceilings for artifact generation,
- defining when an output becomes a reusable runtime capability,
- defining recovery rules for failed or noisy output experiments.

## Anti-Goal

Do not confuse this track with a manual side project where maintainers personally
build all owner-facing UX or creative artifacts themselves.

If that happens, the architecture is compensating for missing runtime capability
instead of developing it.
