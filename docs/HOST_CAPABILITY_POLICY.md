# Host Capability And Self-Improvement Policy

Last updated: 2026-03-28 UTC

## Purpose

This document defines what the `eeepc` system may inspect, change, create, or
propose while running on the host.

The goal is to enable real self-improvement and host adaptation without allowing
unbounded or non-recoverable behavior.

Primary references:

- `docs/PROJECT_CHARTER.md`
- `docs/SOURCE_OF_TRUTH_AND_PROMOTION_POLICY.md`
- `docs/HOST_GITHUB_SYNC_ARCHITECTURE.md`

## Policy Goals

- preserve host safety,
- preserve canonical source ownership,
- allow bounded self-improvement,
- make runtime capabilities inspectable and explainable,
- ensure important changes leave evidence,
- keep operator review in the loop for risky actions.

## Maintainer And Runtime Boundary

This policy governs what the running system may do.

Maintainers define the policy surface, mutation zones, recovery model, and
approval rules.

The running system operates inside that boundary.

Owner-facing outputs should increasingly be generated by the runtime system, not
manually produced by maintainers as a substitute for missing runtime capability.

## Capability States

Every host-facing capability should be represented as one of the following:

- `available` - live and allowed now,
- `blocked_by_policy` - technically possible but intentionally disallowed,
- `unavailable` - not installed, not wired, or not functional,
- `unverified` - not yet checked in the current runtime,
- `degraded` - available with reduced guarantees or temporary constraints.

Free-form operator answers should use the same logic as explicit capability
status commands.

## Capability Classes

### Class A - Read-only host introspection

Default posture:

- generally allowed.

Examples:

- reading system information,
- checking service status,
- inspecting process/resource state,
- reading allowlisted runtime logs,
- reading own code, prompts, config, and reports,
- enumerating available devices and network surfaces.

Requirements:

- avoid secrets exposure,
- respect explicit path restrictions,
- record evidence when introspection drives a significant improvement decision.

### Class B - Bounded workspace mutation

Default posture:

- allowed only in explicit allowlisted surfaces.

Examples:

- prompts,
- selected config files,
- approved code paths,
- helper scripts,
- diagnostics,
- local documentation,
- report/evidence generation logic.

Requirements:

- mutation scope must be bounded,
- changes must be attributable to a cycle/task,
- important changes should emit evidence,
- canonical promotion remains separate.

### Class C - Tool creation and refinement

Default posture:

- allowed when bounded and justified.

Examples:

- host inspection tools,
- parser/research tools,
- optimization helpers,
- small owner-facing utilities,
- creative artifact generators,
- device adapters with safe scope.

Requirements:

- tool purpose must be clear,
- resource cost should fit weak-host constraints,
- tools should be testable or at least evaluable,
- risky integrations require review before broad enablement.

### Class D - Device and sensor interfaces

Default posture:

- review-gated until explicitly enabled.

Examples:

- camera capture,
- microphone capture,
- Bluetooth interactions,
- Wi-Fi/network scanning beyond basic diagnostics,
- media ingestion from external hardware.

Requirements:

- explicit policy enablement,
- visible operator-facing disclosure,
- bounded storage and retention,
- no silent background capture.

### Class E - System-level operational changes

Default posture:

- proposal-first, review-gated.

Examples:

- service definitions,
- boot/startup behavior,
- package installation,
- system configuration changes,
- permission changes,
- network exposure changes.

Requirements:

- generate proposal and rationale first,
- record expected impact and rollback path,
- do not apply blindly during normal autonomous cycles.

### Class F - Canonical source promotion

Default posture:

- always review-gated.

Examples:

- commits to canonical repos,
- promotion branches,
- PR-ready patch bundles,
- repo creation intended to become enduring product/project surface.

Requirements:

- evidence must exist,
- source/evidence ownership boundaries must be preserved,
- target repo/branch must comply with governance rules.

## What The System May Read

The system may read, subject to runtime path/tool policy:

- its own code,
- prompts,
- config,
- tests,
- local docs,
- cycle reports,
- bounded runtime state,
- simulator files,
- service status and host diagnostics,
- allowlisted hardware/device metadata.

It should avoid reading or exporting:

- secrets,
- credential stores,
- unrelated private user material,
- raw sensitive communication archives unless explicitly needed and allowed.

## What The System May Change

The system may change only bounded and policy-allowed surfaces such as:

- its own prompts and prompt layers,
- selected config and runtime settings,
- approved code files inside explicit mutation zones,
- helper scripts,
- tests,
- local docs and runbooks,
- evidence and reporting surfaces,
- owner-facing utilities and dashboards in allowed project surfaces.

The system should not silently change:

- system-wide OS policy,
- security posture,
- external account credentials,
- canonical repo ownership settings,
- high-risk host services,
- unrestricted files outside explicit bounds.

## What The System May Create

The system may create:

- new helper tools,
- reports and cycle artifacts,
- bounded docs,
- tests,
- dashboards and terminal interfaces,
- visualization or creative artifacts,
- prototype games or graphics experiments,
- promotion candidates,
- seed manifests for future projects.

Creation is favored when:

- repeated need is visible,
- owner utility is clear,
- host cost is acceptable,
- outputs can be evaluated and improved.

## Owner Utility Policy

Self-improvement should increase owner value, not only internal optimization.

The system is encouraged to create or refine:

- terminal dashboards,
- workflow helpers,
- research summaries,
- visualizations,
- useful small apps,
- artistic or demoscene-style outputs,
- experiments that teach the system what the owner values.

These outputs must still follow bounded-resource and bounded-scope rules.

## Autonomy Maturity Levels

### Level 0 - bootstrap only

- operator control and introspection only,
- no meaningful self-change beyond essential setup.

### Level 1 - bounded self-maintenance

- prompts, bounded config, docs, and evidence surfaces may evolve,
- system-level changes remain review-gated.

### Level 2 - bounded tool growth

- the system may create and refine lightweight tools inside allowed surfaces,
- host-fit and resource cost must remain explicit.

### Level 3 - runtime-generated owner outputs

- the system may generate owner-facing tools, interfaces, and evaluable artifacts,
- generation should remain bounded, reviewable, and affordable on the target host.

### Level 4 - broader autonomous expansion

- the system may propose or initiate larger project splits or deeper capability
  growth where policy permits,
- canonical promotion and high-risk host changes remain review-gated.

Advancing in maturity should depend on real evidence, not aspiration alone.

## Evaluation Rule For New Artifacts

When creating a new tool, interface, or creative artifact, the system should try
to evaluate it through some combination of:

- direct operator feedback,
- repeated use,
- measurable utility,
- runtime cost,
- maintainability,
- novelty or delight,
- compatibility with weak-host constraints.

Artifacts that are expensive and unused should not accumulate indefinitely.

## Promotion Rule

Host-local creation or mutation does not automatically become canonical source.

Promotion should happen only when:

1. the change proves useful,
2. evidence exists,
3. scope and risk are understood,
4. recovery or rollback is clear,
5. review/policy accepts promotion.

## Required Evidence For Significant Self-Improvement

When the system performs meaningful self-improvement, it should capture at least:

- why the change was attempted,
- what was changed,
- expected benefit,
- observed result,
- resource impact if relevant,
- whether follow-up or promotion is recommended.

## Design Rule For Weak Hardware

No new capability should be treated as complete until it answers all of these:

1. can the target host afford it,
2. can it fail safely,
3. can it be disabled cleanly,
4. can it be explained to the operator,
5. does it create value greater than its maintenance cost.

## Default Decision Rule

If a capability is ambiguous, expensive, privacy-sensitive, or difficult to
recover from:

- prefer `unverified` or `blocked_by_policy`,
- gather evidence,
- produce a proposal,
- wait for review before expanding scope.
